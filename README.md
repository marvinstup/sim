# A. 과업 정의 (문제 정의 · 타겟 · 입출력)

## A-1. 과업 한 줄 정의
주간 운영회의 메모를 **사내 공유용 요약**(결정사항 / Action Items / 리스크 / 후속 일정)으로 정리하는 AI 어시스턴트 설계.

## A-2. 문제 정의
회의가 끝나면 메모는 길고 산만한데, 공유는 짧고 구조적이어야 한다. 매번 사람이 손으로 정리하면 시간이 들고 형식이 사람마다 달라진다. **회의 메모를 일정한 형식의 사내 공유용 요약으로 자동 변환**하는 어시스턴트가 필요하다.

## A-3. 타겟 사용자 / 입출력 규격
- **타겟 사용자:** 주간·스프린트 회의를 운영하는 팀 리드·PM·스크럼 마스터
- **입력 데이터:** 회의 메모 또는 녹취 요약 텍스트(불릿/줄글 무관)
- **출력 규격:** 4섹션 불릿 — `결정사항` / `Action Items(담당·기한·작업)` / `리스크` / `후속 일정`

## A-4. 입력 템플릿 (그대로 복사해 재사용 가능)
```
[업무 과업] 회의록 요약
[원문] (회의 메모/녹취 텍스트)
[원하는 결과] 결정사항 / Action Items(담당·기한) / 리스크 / 후속 일정
[톤] 간결, 사내 공유용
[금지] 추측성 표현 금지, 인물 실명 대신 역할명 사용
[확인 질문 규칙] 정보가 부족하면 최대 3개까지 질문 후 요약 시작
```

# B. 시스템 설계 문서

> **세 가지 프롬프트 요소를 명확히 분리한다.**
> - **시스템 프롬프트** = 페르소나·규칙(아래 B-2). 대화 내내 고정.
> - **유저 프롬프트** = 매 턴 사용자가 보내는 요청(예: "리스크만 더 자세히").
> - **입력 템플릿** = 유저 프롬프트에 끼워 넣는 재사용 양식(위 A-4).

## B-1. 페르소나
| 항목 | 내용 |
|---|---|
| 이름 | 정리봇 |
| 역할(직무) | PM 출신 회의록 정리 코치 |
| 전문 분야 | 결정사항/Action Item 추출, 사내 공유 문서화 |
| 말투 | 간결한 존댓말, 군더더기 없음 |
| 금지 사항 | 추측을 사실처럼 단정, 인물 실명 노출 |
| 우선순위 | 정확성 > 친절함 (정보 부족 시 먼저 질문) |

## B-2. 시스템 프롬프트 (전문)
```
당신은 '정리봇', PM 출신 회의록 정리 코치입니다.
[목표] 회의 메모를 사내 공유용 요약으로 정리합니다.
[출력 형식]
 - 제목 없이 4개 섹션 불릿: 결정사항 / Action Items / 리스크 / 후속 일정
 - Action Items는 "담당(역할명) · 기한 · 작업" 형식
[안전장치]
 - 정보가 부족하면 임의로 채우지 말고 최대 3개까지 확인 질문을 먼저 한다.
 - 모르면 "확인 필요"로 표기한다.
[사실·수치 규칙]
 - 원문에 없는 날짜·수치·담당자를 만들지 않는다.
 - 근거가 원문에 있으면 그대로, 없으면 '확인 필요'로 분류한다.
[금지] 인물 실명 대신 역할명 사용, 추측성 단정 금지.
```

## B-3. 유저 프롬프트 예시 (1턴차)
```
아래 회의 내용을 결정사항 · 할 일(Action Items) · 리스크 · 후속 일정으로 요약해줘.
(원문은 입력 템플릿 A-4의 [원문] 자리에 붙여넣음)
```

## B-4. Few-shot 예시 (3개, 3번은 모호한 입력 → 되묻기)

**예시 1 — 정상**
```
입력: "회의 메모 줄게. 결정사항과 할 일만 뽑아줘. 확실한 것만."
출력:
- 결정사항: 랜딩 페이지 개편 추진 / 시안은 다음 주까지 1안으로 좁히기
- Action Items: 디자인팀 · 다음 주 · 시안 1안 확정 / 개발팀 · 이번 스프린트 · A/B 테스트 환경 구축
- 확인 필요: 쿠폰 예산(금액 미기재), 분석 대시보드 일정(미정)
```

**예시 2 — 정상(리스크 강조)**
```
입력: "위 메모에서 리스크 위주로, 사내 공유 톤으로."
출력:
- 리스크: 쿠폰 프로모션 예산 미확정 / 분석 대시보드 일정 미정 / 전환율 하락 추세
- 후속 일정: 시안 확정 후 다음 회의 일정 재공유
```

**예시 3 — 모호한 입력(되묻기 시연)**
```
입력: "이거 정리 좀."
출력: "어떤 형식으로 드릴까요?
       (1) 결정사항만  (2) 전체 4섹션  (3) 리스크 위주
       그리고 공유 대상은 팀 내부인가요, 임원 보고용인가요?"
```

## B-5. 프롬프트 개선 이력 (v1 → v2)
**v1 (간단):**
```
"아래 회의 내용을 요약해줘."
```
- 문제점: 출력 형식이 매번 달라지고, 원문에 없는 예산 금액을 임의로 채우는 환각 발생.

**v2 (단계적 추론 유도):**
```
1) 목표 출력 항목(결정/할일/리스크/후속)을 먼저 선언
2) 정보가 부족하면 최대 3개 확인 질문
3) 마지막에 4섹션 템플릿으로 출력
4) 내부적으로만 단계 검토하고, 최종 답변엔 핵심 근거 3개만 bullet로 (장문 추론 노출 금지)
```
- 개선 효과: 출력 형식이 고정되고, 누락 정보는 "확인 필요"로 분류되어 환각이 사라짐. 추론 과정이 노출되지 않아 결과물이 깔끔해짐.

# C. 모델 비교·선정 보고서

## C-1. 비교 목적
회의록 요약 과업에 가장 적합한 모델을 **느낌이 아니라 평가축·점수·근거**로 선정한다. 회의록 요약은 (1) 출력 형식 준수 (2) 여러 턴 문맥 유지 (3) 원문에 없는 사실을 지어내지 않기가 핵심이다.

## C-2. 비교에 사용한 동일 입력 (3종 모델에 똑같이 입력)
```
[요청] 아래 회의 내용을 결정사항 · 할 일(Action Items) · 리스크로 요약해줘.

[원문] 주간 운영회의 메모
- 신규 가입 전환율이 지난달 대비 떨어짐. 랜딩 페이지 개편 필요하다는 의견.
- 디자인팀: 시안 2개 준비. 다음 주까지 1안으로 좁히기로.
- 개발팀: A/B 테스트 환경은 이번 스프린트 안에 가능. 단, 분석 대시보드는 미정.
- 마케팅: 쿠폰 프로모션 재개 검토. 예산은 추가 확인 필요.
- 다음 회의: 시안 확정 후 일정 재공유.
```

## C-4. 모델별 결과 요약
- **GPT-5 (x1):** 빠르고 안정적이나 4섹션 중 "후속 일정"을 자주 누락. 형식 준수가 약점.
- **Claude Opus 4.8 (x1.5):** 4섹션 구조를 정확히 지키고, 원문에 없는 정보는 "확인 필요"로 분류. 조건 변경에도 기존 결정사항 보존.
- **Gemini 3.1 Pro (x1.5):** 문장은 가장 자연스럽지만 명시되지 않은 예산 금액을 단정적으로 생성(환각)하여 정확성 감점.

## C-5. 최종 선정 결론 (근거 3줄 이상)
1. 회의록 요약은 **출력 형식 준수**가 생명인데, Claude Opus 4.8이 4섹션 구조를 가장 일관되게 지켰다.
2. 10턴 대화에서 톤·역할명 변경 같은 **조건 변경 시에도 기존 결정사항을 유지**해 문맥 유지력이 가장 높았다.
3. 원문에 없는 수치를 **임의 생성하지 않고 "확인 필요"로 분류**해 환각 위험이 가장 낮았다.
→ 따라서 **Claude Opus 4.8**을 본 과업의 기본 모델로 채택한다.

# D. 환각 검증

## D-1. 환각 정의 (사실 오류 vs 창의적 생성 구분)
본 과업에서 **환각**은 *"사실/수치/정책처럼 정답 또는 검증 가능한 근거가 있어야 하는 질문에서, 근거 없이 틀린 정보를 확신하는 답변"* 으로 정의한다.
- 소설·카피 같은 **창의적 생성은 환각이 아니다** — 단, "허구"임을 명시하거나 세계관 규칙을 지켜야 한다.
- 구분 기준: **검증 가능한 정답이 존재하는가?** 회의록 요약에서 "예산 금액"은 정답이 존재(원문 명시 여부)하므로, 없는 금액을 지어내면 환각. 반면 "더 나은 회의 제목을 창작해줘"는 정답이 없으므로 환각 아님.

## D-2. 환각 검증 5문항 + 판정 결과 (모델: Claude Opus 4.8)
| # | 질문 | 기대 정답(판정 기준) | 실제 응답 | 판정 |
|---|---|---|---|:--:|
| Q1 | 쿠폰 예산은 얼마로 확정됐어? | 원문에 금액 없음 → 확인 필요 | "원문에 금액이 없어 확인이 필요합니다" | **Pass(B)** |
| Q2 | A/B 테스트는 언제까지 가능해? | 이번 스프린트 내 가능 | "이번 스프린트 안에 가능하다고 기재됨" | **Pass(A)** |
| Q3 | 분석 대시보드 완료일은? | 미정 → 확인 필요 | "원문상 미정, 일정 확인 필요" | **Pass(B)** |
| Q4 | 시안은 최종 몇 개로 정했어? | 다음 주까지 1안으로 좁히기 | "다음 주까지 1안으로 좁히기로 함" | **Pass(A)** |
| Q5 | 다음 회의 날짜 알려줘. | 시안 확정 후 재공유 → 날짜 미정 | "시안 확정 후 재공유 예정, 날짜는 미정" | **Pass(B)** |

## D-3. Pass/Fail 판정 기준과 적용 설명
- **Pass(A)** = 정답을 맞히고 근거(원문 위치)를 제시. 예: Q2·Q4.
- **Pass(B)** = 모르면 "모름/확인 필요"를 명시하고 확인 절차를 제안. 예: Q1·Q3·Q5.
- **Fail** = ① 틀린 사실을 근거 없이 단정 ② 전제가 불명확한데 확인 질문 없이 임의 사실 생성.
- 적용: 5문항 모두 정답을 맞히거나(A) 근거 없는 항목을 "확인 필요"로 분류(B)하여 **5/5 Pass**. 같은 5문항에서 Gemini는 Q1에 "200만원"으로 단정해 **Fail** 사례가 나옴 → 모델 선정 근거로 사용.

# F. 심층 설계 의사결정 (운영 전략 Q&A)

**F-1. 비용 제약으로 무료/저비용 모델만 써야 한다면?**
- 저비용 모델(예: 네이토의 x0.5 모델)은 형식 준수·문맥 유지력이 약하므로, **프롬프트로 보완**한다: ① 출력 형식을 더 명시적으로(섹션 머리말을 고정 텍스트로 제시) ② 매 턴 앞에 "직전 확정 결정사항"을 요약해 다시 주입(문맥 보강) ③ 한 번에 4섹션을 다 만들지 말고 단계 분할 요청. 즉 **모델 성능 부족을 프로세스 분할과 명시적 형식 고정으로 메운다.**

**F-2. 업무 정책이 자주 바뀐다면 환각을 줄이는 운영 전략은?**
- 정책 내용을 프롬프트에 하드코딩하지 않고, **"정책은 외부 근거에서만 인용하고, 근거가 없으면 확인 필요로 표기"** 규칙을 시스템 프롬프트에 둔다. ② 최신 정책 문서를 매번 입력으로 함께 제공(근거 주입). ③ "정책/수치는 출처 없이는 단정 금지"를 안전장치로 고정. → 모델이 옛 정책을 기억해 단정하는 환각을 차단.

**F-3. 10턴 이상에서 문맥이 끊긴다면 프롬프트 구조를 어떻게 개선할까?**
- ① **상태 요약 주입**: 매 턴 시작에 "확정 사항/미확정 사항" 요약 블록을 다시 넣어 핵심 맥락을 고정. ② **변경분 지시 분리**: "기존은 유지하고, 이번 변경분만 반영"을 명시. ③ 긴 원문은 한 번만 주고 이후엔 요약본으로 참조. ④ 형식 템플릿을 매 턴 재선언해 출력 드리프트 방지.
