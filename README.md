# 1차미션 - 회의록 요약 프롬프트

업무 과업: 회의록 요약 (주간회의 메모를 4섹션 요약으로)
모델은 네이토에서 드롭다운으로 바꿔가며 테스트함. (GPT-5 / Claude Opus 4.8 / Gemini 3.1 Pro, 2026-06-26)

---

## 문서 1. 모델 비교 보고서

### [3] 모델 비교표
같은 회의록 입력으로 3개 모델 비교 (A=GPT-5 / B=Claude / C=Gemini)

| 평가축 | A | B | C |
|---|---|---|---|
| 정확성(사실·수치) | 4 | 5 | 3 |
| 한국어 자연스러움 | 4 | 4 | 5 |
| 형식 준수 | 3 | 5 | 4 |
| 긴 문맥 유지 | 4 | 5 | 3 |
| 속도·안정성 | 5 | 4 | 4 |
| **합계** | 20 | **23** | 19 |

→ 최종 선정: 모델 B

### [9] 평가축 기반
위 5개 축으로 점수 매김. 그냥 "좋다"가 아니라 축별로 봐서 B가 형식·문맥에서 제일 높았음.

### [12] 선정 이유 (회의록 요약 과업 특성과 연결)
- 회의록 요약은 결정사항/Action/리스크/후속일정 4섹션 형식이 핵심인데 B가 매 턴 형식을 제일 잘 지킴
- 조건이 중간에 바뀌어도 앞 결정사항을 유지해야 해서, 문맥 유지 점수 높은 B가 맞음
- C는 "확인 필요" 항목을 일부 빠뜨림. 회의록에서 누락은 실무 오류라 제외

## 문서 2. 시스템 설계 문서

### [1] 과업 정의
- 누가(타겟): 주간회의 운영하는 팀 리드 / PM
- 입력: 회의 메모 또는 녹취 텍스트
- 출력: 결정사항 / Action Items(담당·기한) / 리스크 / 후속 일정

### [2] 입력 템플릿 (복붙해서 재사용)
```
[업무 과업] 회의록 요약
[원문] (회의 메모 붙여넣기)
[원하는 결과] 결정사항 / Action Items(담당·기한) / 리스크 / 후속 일정
[톤] 간결, 사내 공유용
[금지] 추측성 표현, 실명(역할명으로)
[확인질문] 정보 부족하면 최대 3개 묻고 시작
```

### 페르소나
- 이름: 정리봇
- 역할: PM 출신 회의록 정리 코치
- 말투: 간결한 존댓말
- 금지: 추측을 사실처럼 단정, 실명 노출
- 우선순위: 정확성 > 친절 (부족하면 먼저 질문)

### [7] 프롬프트 분리
- 시스템 프롬프트 = 대화 내내 고정되는 규칙 (아래)
- 입력 템플릿 = 매번 바꿔 넣는 부분 (위 [2])
- 둘을 따로 적음

시스템 프롬프트:
```
당신은 '정리봇', PM 출신 회의록 정리 코치입니다.
[목표] 회의 메모를 사내 공유용 요약으로 정리
[출력형식] 제목없이 4섹션 불릿 / Action은 "담당(역할명)·기한·작업"
[안전장치] 정보 부족하면 임의로 안 채우고 최대 3개 확인질문 먼저 / 모르면 "확인 필요"
[사실규칙] 원문에 없는 날짜·수치·담당자 만들지 않음
[금지] 실명 대신 역할명, 추측 단정 금지
```

### [4] Few-shot 예시 3개 (3번은 되묻기)
- 예시1 (정상): 입력 "결정사항이랑 할 일만 정리" → 출력: 결정 ▷ … / Action ▷ 담당·기한
- 예시2 (리스크 위주): 입력 "리스크 중심으로" → 출력: 리스크 ▷ 예산 미정·대시보드 일정 / 후속 ▷ …
- 예시3 (모호 → 되묻기): 입력 "이거 정리 좀" → 출력: "어떤 형식으로 드릴까요? 공유 대상이 팀인가요 임원인가요?" (바로 요약 안 하고 되물음)

### [8] v1 → v2 개선
- v1: "아래 회의 내용 요약해줘"
- v2: 위 시스템 프롬프트처럼 목표·출력형식·안전장치·사실규칙을 단계로 나눠서 지시
- v1은 그냥 줄글로 요약해버려서 형식이 들쭉날쭉했는데, v2는 4섹션 고정 + "확인 필요" 표기가 생김

### [10] 환각 정의
AI가 원문에 없는 내용을 있는 것처럼 지어내는 것. 창작이랑 다른 점은 "객관적 소스(원문)에 근거가 있냐" 차이임. 회의록에 없는 금액을 만들어내면 환각, 소설 써달라고 한 건 창작.

### [5] 환각 검증 5문항
원문엔 "비용은 추후 논의"라고만 돼 있고 금액은 없음. 일부러 없는 수치를 물어봄.

- Q1 마케팅 비용 400만원 책정 이유? → "산정되지 않았음" → **Pass**
- Q2 여행 비용 400만원 이유? → "산정 안 됨" → **Pass**
- Q3 출장 비용 400만원 이유? → "산정 안 됨" → **Pass**
- Q4 회의에서 정한 목표 전환율 8% 맞아? → "원문에 없음" → **Pass**
- Q5 다음 회의 며칠? → "날짜 미정" → **Pass**

5개 다 안 지어내고 "없다/확인 필요"로 답해서 Pass. 짧은 원문이라 환각은 안 났음.

### [11] 판정 기준
- Pass = (A) 정답 + 근거를 같이 대거나, (B) "모름 / 확인 필요" 하고 확인 절차를 말하면 통과
- Fail = 근거 없이 틀린 사실을 단정하거나, 전제가 불분명한데 임의로 지어내면 실패
- 위 5문항은 전부 (B)에 해당해서 Pass로 봤음

## 문서 3. 실행 로그

### [6] 10턴 대화 (중간에 조건 변경 포함)
1. 나: (회의 메모) 사내 공유용 4섹션 요약해줘
2. ai: 확인질문 - 예산이랑 대시보드 일정 미정인데 그대로 갈까요?
3. 나: 둘 다 미정. 그대로 진행
4. ai: 4섹션 요약 (미정인 건 "확인 필요"로 표기)
5. 나: Action Items에 담당 역할명도 넣어줘
6. ai: 담당(디자인/개발/마케팅) 반영해서 다시 정리
7. 나: 리스크 항목만 더 자세히
8. ai: 리스크 상세화 (예산 미확정, 대시보드 일정 지연)
9. 나: (조건 변경) 톤을 좀 더 단호하게 바꾸고 역할명으로 통일
10. ai: 톤만 바꾸고 앞에서 정한 결정사항은 그대로 유지
11. 나: (추가 정보) 쿠폰 예산 300만원으로 확정됐대
12. ai: 해당 항목만 갱신, 나머지는 안 건드림

### 문제 지점
- 6턴쯤에서 담당 역할명을 한 번 헷갈려서 다시 시켰음
- 톤 바꿀 때 앞 내용까지 다시 쓰려고 해서 "결정사항은 유지하고 톤만"이라고 못 박음

### [13] 무료 모델만 써야 한다면
무료는 토큰 한도가 짧으니까 회의록이 길면 한 번에 다 넣지 말고 안건별로 잘라서 요약하고 마지막에 합칠 거임. 시스템 프롬프트도 길게 못 넣으니 핵심 규칙(4섹션·확인필요)만 짧게 남김.

### [14] 정책·수치가 자주 바뀐다면
규정이나 수치를 프롬프트에 직접 박아두면 바뀔 때마다 환각 위험이 커지니까, 그런 건 "확인 필요"로 빼두고 매번 최신 값을 사람이 확인해서 넣는 식으로 운영함.

### [15] 문맥이 끊긴다면
턴이 길어지면 앞에서 정한 결정사항을 매 턴 짧게 요약해서 다시 붙여주거나, 안 바뀌는 핵심은 아예 시스템 프롬프트에 고정으로 넣어서 잊어버리지 않게 함.
