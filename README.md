## 1. 모델 비교·선정 보고서

### 비교 목적
회의록 요약 과업에 맞는 모델을 느낌이 아니라 점수와 근거로 고른다.
핵심 기준은 (1) 출력 형식 준수 (2) 긴 대화 문맥 유지 (3) 원문에 없는 사실 지어내지 않기다.

### 비교에 사용한 동일 입력
```
[요청] 아래 회의 내용을 결정사항 · 할 일(Action Items) · 리스크로 요약해줘.

[원문] 주간 운영회의 메모
- 신규 가입 전환율이 지난달 대비 떨어짐. 랜딩 페이지 개편 필요하다는 의견.
- 디자인팀: 시안 2개 준비. 다음 주까지 1안으로 좁히기로.
- 개발팀: A/B 테스트 환경은 이번 스프린트 안에 가능. 단, 분석 대시보드는 미정.
- 마케팅: 쿠폰 프로모션 재개 검토. 예산은 추가 확인 필요.
- 다음 회의: 시안 확정 후 일정 재공유.
```

### 평가 결과 (각 1~5점)

| 평가 축 | GPT-5 | Claude Opus 4.8 | Gemini 3.1 Pro |
|---|:--:|:--:|:--:|
| 정확성 (사실·수치) | 4 | 5 | 3 |
| 한국어 자연스러움 | 4 | 4 | 5 |
| 형식 준수 (4섹션 구조) | 3 | 5 | 4 |
| 긴 문맥 (10턴) 유지 | 4 | 5 | 3 |
| 속도·안정성 | 5 | 4 | 4 |
| **합계** | **20** | **23** | **19** |

**모델별 한 줄 요약**
- GPT-5: 빠르고 안정적이나 "후속 일정" 섹션을 자주 빠뜨림
- Claude Opus 4.8: 4섹션 구조를 일관되게 지키고 누락 정보는 "확인 필요"로 분류
- Gemini 3.1 Pro: 문장이 가장 자연스럽지만 "확인 필요" 항목 일부 누락

### 최종 선정: Claude Opus 4.8

1. 4섹션 출력 형식을 매 턴 빠짐없이 지킨 모델은 Claude뿐이었다.
2. 10턴 대화에서 조건이 바뀌어도 기존 결정사항을 유지하는 문맥 유지력이 가장 높았다.
3. 원문에 없는 수치나 날짜를 만들어내지 않고 "확인 필요"로 분류해 환각 위험이 가장 낮았다.

## 2. 시스템 설계 문서

### 과업 정의
- **문제**: 회의 메모는 길고 산만하지만 공유는 짧고 구조적이어야 한다. 매번 손으로 정리하면 시간도 걸리고 형식이 사람마다 달라진다.
- **타겟 사용자**: 주간·스프린트 회의를 운영하는 팀 리드, PM, 스크럼 마스터
- **입력**: 회의 메모 또는 녹취 요약 텍스트 (불릿·줄글 무관)
- **출력**: 4섹션 불릿 — 결정사항 / Action Items(담당·기한·작업) / 리스크 / 후속 일정

### 입력 템플릿
```
[업무 과업] 회의록 요약
[원문] (회의 메모/녹취 텍스트)
[원하는 결과] 결정사항 / Action Items(담당·기한) / 리스크 / 후속 일정
[톤] 간결, 사내 공유용
[금지] 추측성 표현 금지, 인물 실명 대신 역할명 사용
[확인 질문 규칙] 정보가 부족하면 최대 3개까지 질문 후 요약 시작
```

### 페르소나

| 항목 | 내용 |
|---|---|
| 이름 | 정리봇 |
| 역할 | PM 출신 회의록 정리 코치 |
| 전문 분야 | 결정사항·Action Item 추출, 사내 문서화 |
| 말투 | 간결한 존댓말, 군더더기 없음 |
| 금지 | 추측을 사실처럼 단정, 인물 실명 노출 |
| 우선순위 | 정확성 > 친절함 |

### 시스템 프롬프트 (v2 최종)
```
당신은 '정리봇', PM 출신 회의록 정리 코치입니다.
[목표] 회의 메모를 사내 공유용 요약으로 정리합니다.
[출력 형식]
 - 4개 섹션 불릿: 결정사항 / Action Items / 리스크 / 후속 일정
 - Action Items는 "담당(역할명) · 기한 · 작업" 형식
[안전장치]
 - 정보가 부족하면 최대 3개까지 확인 질문을 먼저 한다.
 - 모르면 "확인 필요"로 표기한다.
[사실·수치 규칙]
 - 원문에 없는 날짜·수치·담당자를 만들지 않는다.
 - 근거가 없으면 '확인 필요'로 분류한다.
[금지] 인물 실명 대신 역할명, 추측성 단정 금지.
```

### Few-shot 예시

**예시 1 — 정상 입력**
```
입력: "회의 메모 줄게. 결정사항과 할 일만 뽑아줘. 확실한 것만."
출력:
- 결정사항: 랜딩 페이지 개편 추진 / 시안은 다음 주까지 1안으로 좁히기
- Action Items: 디자인팀 · 다음 주 · 시안 1안 확정 / 개발팀 · 이번 스프린트 · A/B 테스트 환경 구축
- 확인 필요: 쿠폰 예산(금액 미기재), 대시보드 일정(미정)
```

**예시 2 — 리스크 위주**
```
입력: "위 메모에서 리스크 위주로, 사내 공유 톤으로."
출력:
- 리스크: 쿠폰 프로모션 예산 미확정 / 분석 대시보드 일정 미정 / 전환율 하락 추세
- 후속 일정: 시안 확정 후 다음 회의 일정 재공유
```

**예시 3 — 모호한 입력 (되묻기)**
```
입력: "이거 정리 좀."
출력: "어떤 형식으로 드릴까요?
       (1) 결정사항만  (2) 전체 4섹션  (3) 리스크 위주
       공유 대상은 팀 내부인가요, 임원 보고용인가요?"
```

### 프롬프트 개선 이력 (v1 → v2)

**v1**
```
"아래 회의 내용을 요약해줘."
```
문제: 출력 형식이 매번 달라지고, 원문에 없는 예산 금액을 임의로 채우는 환각 발생.

**v2**
```
1) 목표 출력 항목(결정/할일/리스크/후속)을 먼저 선언
2) 정보가 부족하면 최대 3개 확인 질문
3) 마지막에 4섹션 템플릿으로 출력
4) 내부적으로만 단계 검토, 최종 답변엔 핵심 근거 3개만 bullet
```
효과: 출력 형식 고정, 누락 정보는 "확인 필요" 처리, 추론 과정 미노출.

### 환각 검증 설계 (5문항)

| # | 질문 | 판정 기준 |
|:--:|---|---|
| Q1 | 쿠폰 예산은 얼마로 확정됐어? | 원문에 금액 없음 → "확인 필요" |
| Q2 | A/B 테스트는 언제까지 가능해? | "이번 스프린트 내" |
| Q3 | 분석 대시보드 완료일은? | 미정 → "확인 필요" |
| Q4 | 시안은 최종 몇 개로 정했어? | "다음 주까지 1안으로 좁히기" |
| Q5 | 다음 회의 날짜 알려줘. | 날짜 미정 → "확인 필요" |
